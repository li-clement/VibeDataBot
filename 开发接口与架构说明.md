# VibeDataBot 开发接口与架构说明

## 1. 文档目的

这份文档不是项目宣传材料，而是给后续开发者的“真实开发地图”。

它主要回答四类问题：

1. 当前项目到底由哪些模块组成，谁负责界面，谁负责能力实现。
2. 应用从“用户输入一句话”到“界面展示结果”之间，真实的调用链路是什么。
3. 现有功能哪些已经真实落地，哪些仍然是原型级 mock。
4. 后续如果要新增 PDF 能力、数据清洗能力、数据质检能力或新的可视化步骤，应该从哪里接入。

## 2. 项目当前定位

`VibeDataBot-main` 目前是一个“带真实 PDF 提取入口的可视化原型”。

README 和 `concept/` 中描述的是目标产品蓝图，代码里的真实状态更接近下面这个结构：

- 前端壳子是真实的：Next.js App Router + React + Tailwind CSS + Framer Motion。
- 本地文件系统浏览是真实的：通过 Next.js API 读本机目录、代理本地 PDF 文件。
- PDF 基础提取链路半真实：前端调用 `/api/extract-pdf`，再由 Node.js 拉起 Python CLI，Python 用 PyMuPDF 做基础文本块提取和简单阅读顺序组装。
- 其余“清洗、去重、质检、语料切块、Ray 执行日志、集群作业”等，大部分还是前端 `ExecutionEngine` 内部的模拟执行。

所以这个项目不是“前后端都已经工程化完成”，而是“UI 框架、用户操作路径和一个最小后端能力已经打通，方便后续继续增量接真实能力”。

## 3. 仓库结构总览

根目录里最重要的内容如下：

```text
VibeDataBot-main/
├── README.md
├── concept/                         # 产品愿景、模块蓝图说明，不等同于当前实现
└── ray-data-agent-proto/            # 真正运行的原型工程
    ├── app/                         # Next.js App Router 入口与 API
    ├── features/                    # 按功能切分的前端与 Python 能力模块
    ├── public/                      # 图片资源
    ├── tests/                       # Python 侧简单验证脚本
    ├── package.json                 # 前端依赖与脚本
    └── tsconfig.json
```

如果你是第一次接手这个项目，应该把 `ray-data-agent-proto/` 视为主工程，把 `README.md` 和 `concept/` 视为“产品目标和设计意图参考”。

## 4. 技术栈与运行方式

### 4.1 前端与界面层

- 框架：Next.js 16
- 视图库：React 19
- 样式：Tailwind CSS 4
- 动效：Framer Motion
- 图标：Lucide React
- UI 状态管理：项目内自定义 React Context

前端启动命令：

```bash
cd ray-data-agent-proto
npm run dev
```

默认访问地址：

```text
http://localhost:3000
```

### 4.2 Python 能力层

当前 Python 侧实际使用到的核心库：

- `PyMuPDF` (`fitz`)：PDF 打开、文本块提取
- 标准库 `argparse` / `json`：CLI 包装与 Node/Python 通信

测试脚本里还使用了：

- `fpdf`：生成临时测试 PDF

### 4.3 Node 与 Python 的协作方式

项目没有单独的 Python Web 服务。

当前模式是：

1. 前端 `fetch("/api/extract-pdf")`
2. Next.js Route Handler 用 `child_process.exec` 调 `python3 features/pdf_extractor/logic/cli.py`
3. `cli.py` 调用 `build_pdf_pipeline(file_path)`
4. Python 把 JSON 结果打印到 stdout
5. Next.js 读取 stdout，解析成 JSON，再返回给浏览器

这意味着 Python 能力目前是“按次拉起、命令行式”集成，不是常驻服务。

## 5. 功能分块与职责边界

### 5.1 `app/`：应用入口与本地 API

关键位置：

- `app/page.tsx`：页面入口，只渲染 `AgentInterface`
- `app/layout.tsx`：全局布局、字体、主题壳
- `app/globals.css`：全局主题变量与基础样式
- `app/api/extract-pdf/route.ts`：PDF 提取 API
- `app/api/fs/route.ts`：本地目录浏览 API
- `app/api/file/route.ts`：本地文件代理 API

这一层的职责是：

- 提供浏览器端可访问的页面入口
- 用 Next.js Route Handler 承担“本机资源访问代理”的作用
- 把前端和 Python CLI 串起来

### 5.2 `features/agent/`：交互中枢与执行编排

这是整个原型最核心的前端业务目录。

包含三类角色：

1. `components/`
   - `AgentInterface.tsx`：整体页面骨架
   - `ChatInterface.tsx`：聊天输入框、生成计划、执行计划
   - `MainCanvas.tsx`：主画布，负责消息区、Pipeline、结果区、日志区
   - `MessageList.tsx`：消息列表
2. `context/`
   - `AgentContext.tsx`：全局共享状态
3. `logic/`
   - `PlanGenerator.ts`：把用户输入转成步骤列表
   - `ExecutionEngine.ts`：按步骤顺序执行，并生成 artifact 和日志
4. `types/`
   - `AgentTypes.ts`：`StepType`、`ExecutionPlan`、消息类型等

这一层决定了：

- 前端支持哪些任务类型
- 用户一句话会被解析成哪些步骤
- 结果在主画布上如何显示

### 5.3 `features/navigation/`：左侧导航与本地文件树

关键文件：

- `Sidebar.tsx`
- `FileTree.tsx`

这一层负责：

- 显示项目、数据源、集群等导航项
- 通过 `/api/fs` 浏览本地目录
- 当用户点击本地文件时，把“带路径的提示语”自动写入聊天输入框

特别要注意：

- 只有点击 `.pdf` 文件时，系统会自动注入“提取该 PDF 文件内容，路径是：...”这类提示语。
- 这意味着“本地文件树 -> 聊天输入框 -> 计划生成器识别路径”是一个显式耦合链路。

### 5.4 `features/pipeline/`：计划可视化

关键文件：

- `PipelineVisualizer.tsx`

当前能力：

- 只支持线性步骤展示
- 每个步骤节点显示状态、图标和标签
- 点击节点后，主画布显示代码片段和对应 artifact

限制：

- 还不支持真正的 DAG 分支、汇合、重试、并行节点
- 视觉上是 pipeline，但执行上仍是 `for ... of` 串行

### 5.5 `features/data-view/`：结果展示层

关键文件：

- `MarkdownViewer.tsx`
- `DataFrame.tsx`
- `MockDataService.ts`

职责分为两部分：

1. 展示组件
   - `MarkdownViewer`：PDF 左右对比视图，左侧原 PDF，右侧提取文本
   - `DataFrame`：表格化展示结构数据
2. 原型数据服务
   - `MockDataService`：生成假日志数据、模拟 PII 扫描和清洗

这说明：

- 视图组件本身相对可复用
- 但很多上游数据仍然来自 mock，不是后端真实返回

### 5.6 `features/resources/`：资源详情页

关键文件：

- `ResourceDetailView.tsx`

作用：

- 显示“数据源”“集群”的详情卡片和统计信息

现状：

- 这里的数据是纯前端静态 mock
- 它更多承担“产品愿景展示”和 UI 占位作用，不参与真实执行

### 5.7 `features/pdf_extractor/`：Python PDF 提取能力

这是目前最接近真实后端能力的模块。

目录结构：

```text
features/pdf_extractor/
├── logic/
│   ├── cli.py
│   ├── pipeline.py
│   ├── router.py
│   ├── layout_analyzer.py
│   ├── order_assembler.py
│   └── ocr_engine.py
└── types/
    └── PdfStructures.ts
```

各文件职责如下：

- `cli.py`
  - 命令行入口
  - 接收 `--file_path`
  - 调 `build_pdf_pipeline`
  - 把结果作为 JSON 输出到 stdout

- `pipeline.py`
  - Python 主入口
  - 验证文件路径
  - 打开 PDF
  - 调路由决定走 `FAST_TRACK` 还是 `DEEP_TRACK`
  - 汇总 block
  - 调阅读顺序组装器输出 markdown

- `router.py`
  - 根据第一页是否能提取到足够文本，决定快速通道还是深度通道
  - 当前规则非常简单，只看第一页文本长度

- `layout_analyzer.py`
  - 使用 `page.get_text("blocks")` 提取文本块和图像块
  - 当前并不是视觉模型版面分析，只是 PyMuPDF 的 block 提取

- `order_assembler.py`
  - 根据 `(page, round(y / tolerance), x)` 做启发式排序
  - 把 block 拼成 markdown 文本

- `ocr_engine.py`
  - 当前是 mock，占位深度 OCR 通道
  - 扫描件只会返回固定的占位文本

- `types/PdfStructures.ts`
  - 前端侧对 PDF 提取结果结构的类型声明

## 6. 真实运行链路

### 6.1 从页面加载到主界面展示

页面启动链路：

```text
app/page.tsx
-> AgentInterface
-> AgentProvider
-> Sidebar + MainCanvas + ChatInterface
```

这里的关键点：

- `AgentProvider` 是全局状态容器
- `Sidebar`、`ChatInterface`、`MainCanvas` 都依赖同一个上下文
- 所有执行日志、消息、计划、artifact 都保存在 `AgentContext`

### 6.2 从点击 PDF 到生成执行计划

流程如下：

```text
用户点击 FileTree 中的 PDF
-> FileTree 把“提取该 PDF 文件内容，路径是：xxx.pdf”写入 chatInput
-> 用户发送消息
-> ChatInterface 调用 MockPlanGenerator.generate(userInput)
-> 生成 ExecutionPlan
-> MainCanvas 显示 PipelineVisualizer
```

这条链路说明：

- 文件树本身不直接发起 PDF 提取
- 它只是帮助用户快速构造一个能被 `PlanGenerator` 识别的输入

### 6.3 从执行计划到结果展示

执行流程如下：

```text
用户点击 Execute Plan
-> ChatInterface 调用 ExecutionEngine.executePlan(plan)
-> ExecutionEngine 逐步执行每个 step
-> 更新 step 状态、追加日志、生成 artifact
-> AgentContext 保存结果
-> MainCanvas 根据 artifact 类型渲染 MarkdownViewer 或 DataFrame
```

这里的核心事实：

- 执行顺序是串行的
- artifact 通过 `artifactPayloads` 在步骤之间传递
- 是否展示为 PDF 对比视图，取决于 artifact 中是否包含 `_is_pdf_result`

## 7. 当前已实现接口说明

### 7.1 `GET /api/fs`

用途：

- 浏览本地目录树

请求参数：

- `dir`：目标目录，可选；为空时默认 `os.homedir()`

返回结构：

```json
{
  "path": "/absolute/path",
  "items": [
    {
      "name": "example.pdf",
      "path": "/absolute/path/example.pdf",
      "isDirectory": false
    }
  ]
}
```

说明：

- 会过滤隐藏文件
- 文件夹排前，文件排后
- 当前没有工作区白名单限制，属于本机级目录读取

### 7.2 `GET /api/file`

用途：

- 代理本地文件给浏览器预览
- 当前主要用于 iframe 内预览 PDF

请求参数：

- `path`：本地文件路径

兼容处理：

- 会去掉 `local:` 前缀
- 会去掉 `file://` 前缀

支持内容类型：

- `.pdf`
- `.jpg`
- `.jpeg`
- `.png`

说明：

- 当前直接把整个文件读入内存再返回
- 大文件场景下应该改为流式返回

### 7.3 `POST /api/extract-pdf`

用途：

- 触发 Python PDF 提取能力

请求体：

```json
{
  "filePath": "/absolute/path/to/file.pdf"
}
```

成功返回：

```json
{
  "doc_id": "file.pdf",
  "source_url": "local:///absolute/path/to/file.pdf",
  "markdown_content": "....",
  "metadata": {
    "page_count": 3,
    "fast_track_enabled": true
  },
  "extracted_images": [],
  "_is_scanned_pdf": false,
  "_processing_time_ms": 123.45
}
```

失败返回：

```json
{
  "error": true,
  "message": "..."
}
```

约束：

- Node 侧默认使用 `python3`
- Python CLI 路径是 `features/pdf_extractor/logic/cli.py`
- 结果必须从 stdout 输出纯 JSON

## 8. Python 内部能力接口

### 8.1 `build_pdf_pipeline(file_path: str) -> Dict[str, Any]`

这是 Python 侧最重要的对外接口。

输入：

- `file_path`：本地 PDF 绝对路径或相对路径

输出字段：

- `doc_id`
- `source_url`
- `markdown_content`
- `metadata`
- `extracted_images`
- `_is_scanned_pdf`
- `_processing_time_ms`

当前执行步骤：

1. 校验文件存在
2. `fitz.open(file_path)`
3. `analyze_pdf_and_route(doc)`
4. 如果是 `FAST_TRACK`，逐页 `extract_blocks_from_page`
5. 如果是 `DEEP_TRACK`，调用 `run_deep_ocr_pipeline`
6. `assemble_blocks_to_markdown`
7. 返回结果字典

### 8.2 `analyze_pdf_and_route(doc)`

职责：

- 识别文档是否更适合走“原生文本解析”还是“深度 OCR”

当前策略：

- 只检查第一页
- 提取到的文本长度大于 10，走 `FAST_TRACK`
- 否则走 `DEEP_TRACK`

意味着：

- 这是一个 MVP 级规则，不适合复杂混合文档

### 8.3 `extract_blocks_from_page(page, page_index)`

职责：

- 从单页中提取文本块与图像块

返回 block 结构大致为：

```python
{
    "type": "text" | "image",
    "content": "...",
    "bbox": (x0, y0, x1, y1),
    "page": 0,
    "block_no": 1
}
```

注意：

- 当前“layout analyzer”不是模型推理器
- 它只是从 PyMuPDF 得到 block，再做轻量整理

### 8.4 `assemble_blocks_to_markdown(blocks, y_tolerance=8.0)`

职责：

- 对多页 block 进行近似阅读顺序重排
- 拼装最终 markdown 文本

当前策略：

- 按页码
- 按量化后的 y 坐标
- 再按 x 坐标

适用场景：

- 一般文本型 PDF 的粗粒度排序

局限：

- 复杂双栏、跨栏标题、浮动图表、脚注、公式区域并不可靠

### 8.5 `run_deep_ocr_pipeline(file_path)`

职责：

- 占位“深度 OCR”链路

现状：

- 返回固定 mock 内容
- 没有真实 OCR、版面识别、表格恢复、公式恢复能力

## 9. 前端执行引擎的耦合关系

这一节很重要，因为它直接回答“后续新功能能不能模块化接入”。

### 9.1 当前可视化能力与业务逻辑的耦合程度

结论先说：

- “界面壳”相对可复用
- “功能接入方式”目前仍是显式耦合

具体体现在：

1. 新能力类型要先加到 `AgentTypes.ts` 的 `StepType`
2. `PlanGenerator.ts` 要能把用户输入解析成这个新步骤
3. `ExecutionEngine.ts` 要在 `switch(step.type)` 里实现执行逻辑
4. `PipelineVisualizer.tsx` 要给这个步骤配置图标
5. `MainCanvas.tsx` 需要知道这个步骤产出的 artifact 应该用什么组件展示

也就是说，当前项目并不是“插件式注册新节点后自动出现在界面里”，而是“枚举 + 分支判断式接入”。

### 9.2 现有 PDF 功能与界面的耦合方式

PDF 功能的接入方式是：

```text
FileTree 提供路径
-> PlanGenerator 生成 EXTRACT_PDF 步骤
-> ExecutionEngine 在 EXTRACT_PDF 分支里调用 /api/extract-pdf
-> 返回结果被包装成带 _is_pdf_result 的 artifact
-> MainCanvas 识别后调用 MarkdownViewer
```

所以：

- PDF 提取能力本身可以继续在 Python 模块里独立演进
- 但它如何被前端触发、如何被 pipeline 展示，仍由前端编排层控制

### 9.3 清洗、去重、质检、语料生成和界面的耦合程度

目前这几类能力：

- 不在 Python 独立模块中
- 不通过单独 API 调用
- 不通过真实 Ray 后端运行
- 主要写在 `ExecutionEngine.ts` 的分支逻辑里

因此它们和当前可视化界面是“强耦合原型实现”。

优点：

- 快速演示产品流程
- 前端就能控制全链路表现

缺点：

- 真实能力难以复用
- 测试粒度不清晰
- 替换为后端实现时需要改动多个位置

## 10. 后续开发时的推荐改造方向

### 10.1 如果只是新增一个“可视化展示步骤”

例如新增：

- 文本摘要
- 关键词提取
- 情感分析

可以沿用当前结构，最少需要改这些位置：

1. `features/agent/types/AgentTypes.ts`
2. `features/agent/logic/PlanGenerator.ts`
3. `features/agent/logic/ExecutionEngine.ts`
4. `features/pipeline/components/PipelineVisualizer.tsx`
5. 必要时新增展示组件，并在 `MainCanvas.tsx` 中接入

这种情况下不需要重做整个界面。

### 10.2 如果要把 mock 能力替换成真实后端能力

推荐按下面方式改：

1. 把 `CLEAN_TEXT`、`DEDUPLICATE`、`QUALITY_CHECK`、`GENERATE_CORPUS` 从 `ExecutionEngine.ts` 中抽离
2. 在 `features/pdf_extractor/` 下新增真实 Python 模块，例如：
   - `cleaner.py`
   - `deduplicator.py`
   - `quality_checker.py`
   - `corpus_builder.py`
3. 让 Python 主入口支持传入“步骤链”或“处理配置”
4. Next.js API 只做参数透传和结果回传
5. `ExecutionEngine` 只负责调度 API，不直接持有算法实现

这样做之后：

- 算法能力和界面解耦
- Python 能力可单独测试
- 将来接 Ray、任务队列或独立服务也更自然

### 10.3 如果要支持真正的 DAG / 并行执行

当前 `PipelineVisualizer` 和 `ExecutionEngine` 都默认线性串行流程。

如果未来要支持：

- 分支节点
- 条件节点
- 并行节点
- 重试与中断恢复

就需要重新设计两层：

1. `ExecutionPlan` 数据结构
2. `PipelineVisualizer` 的布局与连线方式

所以答案是：

- “后续新功能可不可以直接集成进去？”  
  大部分线性步骤可以。
- “需不需要重新设计？”  
  如果目标是复杂 DAG、真实调度系统、节点级监控，则需要。

## 11. 建议的增量开发路线

为了避免一次性重构过大，建议按下面顺序推进。

### 阶段 1：把真实能力边界补齐

目标：

- 保留现有 UI，不动交互骨架
- 逐步把 mock 算法迁移到 Python

建议动作：

1. 先补 `cleaner.py`
2. 再补 `deduplicator.py`
3. 再补 `quality_checker.py`
4. 最后补 `corpus_builder.py`

### 阶段 2：把执行协议稳定下来

目标：

- 不再让 `ExecutionEngine` 同时承担“编排 + 算法”

建议动作：

1. 为每个步骤约定统一输入输出结构
2. 统一 artifact 元数据字段
3. 前端只关心步骤状态和结果类型

### 阶段 3：再考虑真实调度

目标：

- 让 README 中提到的 Ray 能力逐步落地

建议动作：

1. 把 Python CLI 替换成独立服务或任务执行器
2. 支持异步任务状态查询
3. 将日志、进度、结果存入统一存储

## 12. 当前代码中的关键限制与风险

### 12.1 安全边界较弱

- `/api/fs` 默认可以浏览用户家目录
- `/api/file` 可以读取任意本地文件
- `/api/extract-pdf` 直接执行本地 Python 命令

这适合本地原型，但不适合直接部署为多人可访问服务。

### 12.2 `extract-pdf` 的命令拼接比较脆弱

当前是直接拼出：

```text
python3 <cli.py> --file_path "<path>"
```

如果路径中包含特殊字符，存在转义与健壮性问题。后续建议改成 `spawn` 或更安全的参数传递方式。

### 12.3 Python 返回协议依赖 stdout 必须是纯 JSON

这意味着：

- Python 业务日志不能直接打到 stdout
- 否则 Node 侧 `JSON.parse(stdout.trim())` 会失败

当前代码已经把部分日志打到 stderr，但后续扩展时需要继续遵守这个约束。

### 12.4 深度 OCR 仍是占位

扫描件、公式、表格、图像理解能力目前都未真实落地。

README 里的这些能力要视为“规划目标”，不能视为当前可用能力。

### 12.5 结果展示的类型判断仍然偏硬编码

主画布主要靠：

- 是否有 `_is_pdf_result`
- 是否存在 `chunk_id`

来决定渲染什么组件。

后续如果结果类型变多，建议引入显式的 artifact schema/type。

## 14. 结论

对这个项目最准确的理解应该是：

- 它已经具备一个可继续扩展的界面骨架
- 也已经打通了“前端 -> Next.js API -> Python CLI -> PDF 提取 -> 前端结果展示”的最小闭环
- 但除 PDF 基础提取外，大部分数据飞轮能力仍处于 mock 或概念验证阶段

因此后续开发不需要推倒重做，但也不能把现状当成“已经模块化完成的生产架构”。

最务实的路线是：

- 保留现有 UI 和交互骨架
- 逐步把 `ExecutionEngine` 里的 mock 能力迁到 Python 或独立服务
- 等后端能力稳定后，再升级 Pipeline 的 DAG 表达能力和任务调度能力

这会比一次性重构整套系统更稳，也更适合当前项目状态。
